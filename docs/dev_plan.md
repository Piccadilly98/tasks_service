# План масштабирования и развития проекта

## Основные изменения(в порядке приоритетов)
### Хранилище данных
Первым делом, после стабильно работающего MVP следует запланировать переход с in-memory хранилища(на оперативной памяти) на Реляционную базу данных, самым оптимальным вариантом в текущее время является `Postgres`, так как:
- Он широко используется в продакшн продуктах
- В отличии от `SQLite` например, Postgres является СУБД, которая работает в рамках клиент-серверной архитектуры, что позволяет нам расположить само хранилище данных на отдельной машине, снизив нагрузку и использование ресурсов нашего сервера
- Понятная документация, в том числе на русском языке. Большое колличество информации в открытом доступе.  
#### Преимущества
1. **Сохранность данных** - при перезапуске сервиса, данные не будут теряться, так как хранилище располагается на жестком диске
2. **Масштабирование**, за счёт клиент-серверной архитектуры posgtres, что позволяет нам расположить хранилище данных на отдельном сервере
3. **Консистентность данных**, за счёт поддержки модели `ACID` и транзакций. Это позволяет нам явно вносить изменения данных полностью, либо не вносить вовсе.
4. Оптимальный расход ресурсов машины - меньший расход оператной памяти(RAM), за счёт переноса всех данных на жесткий диск

### Кэширование
Для сокращения числа запросов к БД и сокращения времени ответа клиенту, следует рассмостреть внедрение кэширования, для хранения востребованных данных, таких как:
- API или JWT токены - для ускорения авторизации и аутентификации пользователей
- Часто запрашиваемые задачи пользователей, для ускорения получения клиентом этой или этих задач
- В случае перехода на сессии, вместо токенов, то следует хранить в кэше токены сессий  
Для добавление кэширования, лучше использовать `NoSQL` хранилище, лучшим примером, которого является `Redis`. Redis является полноценным хранилищем, с моделью `ключ-значение`, которое располлагается на оперативной памяти.
#### Преимущества
1. Быстрый доступ к данным - получение данных в 10–1000 раз(в зависимости от многих факторов)
2. Возможность настройки сохранения данных на жесткий диск, для безопасности при перезапуске
3. Настройка времени жизни данных - `TTL`, по истечению которого данные удаляются, являясь неактуальными
4. Клиент-Серверная архитектура, за счёт которой возможно разместить кэш на отедльной машине(сервере), разгрузив основной

### Веб Фреймворк
Для повышения производительности сервиса и повышения колличества `RPS`(request per second) следует переход на веб-фреймворк, который позволит сервису выдерживать большие нагрузки. Хорошим примером полноценного веб-фреймворка является Fiber, который построен на `собственном движке(fasthhtp)`, позволяющий намного быстрее и оптимальнее обрабатывать запросы.  
#### Преимущества:
1. Рост производительности - Fasthttp переиспользует соединения и оптимизирует работу с памятью, что позволяет обрабатывать значительно больше параллельных запросов.
2. Обширный выбор инструментов - Fiber имеет большое колличество инструментов по работе с соединениямии данными
3. Fiber имеет схожий со стандартной библиотекой интерфейс, что упрощает переход.

### Протоколы общения
Для повышения скорости общения между микросервисами, общающимися по протоколу `HTTP`, который не является бинарным, следует заложить переход на `grpc`.
#### Преимущества:
1. gPRC - является бинарным протоколом, работающим поверх `HTTP/2`, переиспользуется соединения и ускоряет время от сервера клиенту в 7-10 раз в отличии от `HTTP/1*`
2. gRPS активно используется в коммерческих проектах, что позволяет нам изучить все тонкости работы с ним и подобрать удобныый для себя варинат интеграции в проект
3. Сокращение общего времени ответа клиенту, за счёт ускорения общения между микросервисами.
